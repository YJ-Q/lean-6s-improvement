# Lean 6S Improvement Case Study 设计方案

日期：2026-07-19
状态：已完成对话确认，待用户审阅书面方案

## 1. 项目背景

制造现场的管理者通常能够发现工具混放、通道占用、污染、标准缺失等异常，但不一定掌握精益改善的完整方法。他们容易停留在“清理一下”“提醒员工”“加强管理”等一次性处理，难以系统完成风险判断、任务派工、验收和防复发。

本项目尝试把精益改善专家的判断框架封装为可配置的 AI Skill。现场管理者无需学习完整的 Lean 6S 理论，只需在支持 Skill 的智能体应用中完成配置并描述现场问题，即可获得结构化的追问、分类、整改动作、验收标准和闭环证据要求。

Case Study 的主角是 `lean-6s-improvement` Skill 的产品设计与迭代，不是传统制造业咨询项目，也不是验收报告的摘要。

## 2. Case Study 目标

### 2.1 第一目标

服务 AI 产品经理岗位求职，证明作者具备以下能力：

- 从制造业现场问题中识别 AI 产品机会。
- 把专家知识拆解为可执行的产品工作流和规则体系。
- 使用案例、专家反馈、有无 Skill 对照和跨模型测试驱动迭代。
- 正确认识 AI 评测的可信边界，不把案例回归等同于真实世界效果。
- 在 AI 辅助编程条件下主导产品逻辑、验收标准和迭代方向。

### 2.2 辅助目标

体现作者“懂业务也能做 AI 落地”的复合能力，为制造业数字化、AI 应用产品等综合岗位提供证明。

### 2.3 非目标

- 不宣称 AI 可以替代精益改善专家。
- 不宣称已通过大规模真实用户验证。
- 不宣称已经提高整改效率、降低事故率或产生确定业务收益。
- 不把代码复杂度作为项目价值的主要证明。
- 不公开原始验收报告或任何可识别客户的信息。

## 3. 目标受众与使用场景

主要受众是招聘方、AI 产品负责人和面试官。

Case Study 在求职流程中的使用路径为：

1. 简历项目经历建立初步认知。
2. PDF 精简版在 3–5 分钟内建立兴趣。
3. 网页完整版用 8–15 分钟证明项目深度。
4. 项目掌握手册帮助作者准备面试答辩，不对外公开。

内容采用中文主叙事，英文用于项目标题、章节标签和必要的 AI 产品术语。第一版不制作完整中英双语页面；以后投递外企时再基于同一结构制作英文版。

## 4. 作者角色与贡献边界

作者主导以下工作：

- 问题定义和产品机会识别。
- Skill 产品形态选择。
- 工作流、规则体系和输出方式设计。
- Python MVP 与配套文件的 AI 辅助实现和验收。
- 有无 Skill、跨模型和多轮迭代方案。
- 发现问题后的规则修改方向和最终产品决策。

案例整理和部分最终测试由协作者完成。Case Study 必须明确团队贡献，不把协作者完成的工作描述为作者独立完成。

代码主要由 AI 生成。对外表述应为：作者主导产品问题、规则逻辑、验收和迭代，使用 AI 辅助完成代码实现。作者需要通过项目掌握手册和代码练习达到可解释核心实现的程度，不能把自己尚未理解的机制包装为个人技术能力。

## 5. 核心定位与叙事

### 5.1 暂定标题

**Lean 6S Improvement：把精益改善专家方法封装成现场管理者可用的 AI Skill**

### 5.2 一句话介绍

我设计并迭代了一套 Lean 6S AI Skill，将问题分类、风险判断、整改派工、验收和防复发逻辑转化为可复用工作流，帮助缺少精益改善经验的现场管理者系统处理现场问题。

### 5.3 叙事比例

- 60% 产品迭代。
- 25% 评测证据。
- 15% 行业场景。

### 5.4 核心故事

1. 现场管理者不缺少发现问题的能力，缺少系统分析和改善方法。
2. 通用 AI 容易给出泛化建议，难以直接派工、验收或判断真闭环。
3. 作者将精益改善方法封装为 Skill，并通过真实案例、导师评审、有无 Skill 对照和跨模型测试持续发现不足。
4. 产品从“生成长篇分析报告”演进为“台账优先、追问前置、证据闭环”的现场改善助手。
5. 当前原型在 31 条现有案例上完成回归，同时保留尚未规模化用户验证、依赖输入质量和规则硬编码等限制。

## 6. 对外产物

### 6.1 网页完整版

网页是主要作品集载体，用于证明思考深度和完整过程。建议包含九个章节：

1. **Hero**：项目价值、作者角色、周期和结果概览。
2. **Problem**：管理者能发现问题，却缺少系统改善方法。
3. **Opportunity**：为什么选择可配置 Skill，而不是独立 App 或普通提示词。
4. **Solution**：从现场输入到追问、分类、整改、验收和闭环的完整工作流。
5. **Early Failure**：早期版本措施太泛、追问不足、非典型问题边界不稳。
6. **Key Iterations**：安全优先、主副分类、习惯性异常、追问关卡、台账优先和假闭环审核。
7. **Evaluation**：案例库、有无 Skill、跨模型、自动回归和合成用户测试。
8. **Results & Limitations**：当前结果、专家反馈、合成测试发现及可信边界。
9. **Reflection & Next Steps**：真实用户研究、减少硬编码和优化输入体验。

### 6.2 PDF 精简版

PDF 控制在 8–12 页，服务简历附件和招聘平台上传。它不复制全部网页，而是按以下路径重新编排：

> 问题 → 洞察 → 方案 → 关键迭代 → 前后对比 → 评测 → 结果 → 反思

PDF 只保留最能证明 AI 产品能力的决策和证据，并在结尾提供网页链接和二维码。

### 6.3 离线 HTML 包

提供无需登录、安装环境或运行命令即可双击打开的备份版本。离线包不得包含原始验收报告、内部测试原文或敏感附件。

### 6.4 项目掌握手册

掌握手册不公开投递，用于学习和面试准备，包括：

- 产品决策解释。
- Skill 文件逐段解读。
- Python 脚本数据流和核心函数。
- 规则文档、案例库和脚本的关系。
- 硬编码、过拟合和技术债分析。
- 面试问答、三分钟版本和十分钟版本讲稿。
- 代码阅读练习和一次小型规则修改练习。

## 7. 视觉设计

### 7.1 视觉方向

采用 **Claude-inspired 编辑式极简**。借鉴其温暖、克制和重视排版的气质，不复制 Anthropic 页面，不使用 Anthropic 或 Claude Logo，也不把作品伪装为官方设计。

视觉特征：

- 暖象牙白背景、炭黑正文、陶土橙少量强调。
- 中文大标题使用克制的衬线字体，正文使用清晰的无衬线字体。
- 大留白、细分隔线、章节编号和编辑式网格。
- 避免紫色科技渐变、装饰性 Emoji、满屏圆角卡片和无意义统计数字。
- 用真实内容、过程证据和排版层级建立视觉质量。

### 7.2 视觉母题

贯穿页面的视觉母题为：

> 模糊的现场描述，逐步转化为结构化的整改闭环任务。

首页、工作流、迭代和案例模块都围绕这一转化展开，避免使用与项目无关的抽象 AI 装饰图。

### 7.3 关键模块

- 首页使用一句强标题和一张代表性工作流图。
- 关键迭代使用 **Before / Insight / Decision / After** 结构。
- 代码模块允许切换“产品解释 / 伪代码 / Python 实现”。
- 评测模块使用真实数据图表和条件切换。
- 网页提供固定章节导航、案例展开、版本对比和模型测试切换等轻交互。

### 7.4 视觉验证方式

完整制作前先完成一张低成本 HTML 视觉样张，只覆盖首页、一个迭代章节和一个评测模块。样张获得确认后再扩展完整页面。

## 8. 证据体系

### 8.1 证据来源

| 来源 | Case Study 用途 |
| --- | --- |
| 脱敏验收报告 | 行业背景、典型现场问题、专家方法来源 |
| `examples/cases.md` | 虚拟边界案例和规则设计过程 |
| `examples/cases2.md` | 真实项目记录整理后的脱敏案例 |
| 有无 Skill 对比报告 | 追问、风险、结构化和闭环能力差异 |
| 跨模型报告 | Skill 在不同模型上的稳定作用及模型差异 |
| 真实场景优化复测报告 | 导师反馈到追问、多轮更新和现场适配的迭代 |
| Git 提交记录 | 从报告型输出到台账优先的演进过程 |
| Python 回归测试 | 现有案例上的主分类、副分类、习惯性异常和闭环一致性 |
| 合成用户测试 | 输入质量、追问负担和角色适配的形成性问题发现 |

### 8.2 证据标签

所有结果分为三类：

1. **已验证**：31 条现有案例回归通过。
2. **专家反馈**：具有精益改善咨询经验的导师试用后认可方向，并指出需要更契合现场情况；该反馈推动追问关卡等设计。
3. **尚未验证**：没有大规模真实用户试用，不能声称已产生真实业务绩效。

### 8.3 31/31 的正确解释

31/31 证明现有规则、脚本与现有案例标注一致。它不能证明未知现场问题准确率为 100%，原因包括：

- 案例规模有限。
- 案例与规则共同迭代，存在测试集泄漏风险。
- `classify()`、`habitual()` 和 `closure_check()` 存在针对已知表达的硬编码。
- 真实用户可能提供更短、更模糊或带误导的信息。

## 9. 合成用户测试

### 9.1 定位

合成用户测试用于正式招募前的形成性评测，发现交互、输入和规则问题。它必须明确标注为 synthetic user testing，不能写成真实用户满意度、真实用户访谈或市场验证。

### 9.2 测试角色

冻结五张人物卡：

1. 精益新手班组长：能发现问题，但习惯使用模糊口语。
2. 时间紧张的仓库主管：希望快速得到可复制进台账的结果。
3. 风险敏感的 EHS 人员：关注安全边界、证据和责任。
4. 有经验的精益顾问：会挑战分类、原因和闭环标准。
5. 数字工具不熟练的现场管理者：对追问和复杂输出的理解能力较弱。

每张人物卡固定知识水平、表达方式、配合程度、数字工具熟练度和隐藏现场信息。

### 9.3 测试任务

每个角色完成三类任务：

1. 信息相对完整的明确问题。
2. 信息不足、需要关键追问的模糊问题。
3. 用户声称“已整改”、需要审核假闭环的问题。

每个任务分别运行普通 AI 条件和配置 Skill 条件，共约 30 个测试会话。两个条件使用相同基础任务、相同人物卡和相同隐藏信息；任务顺序交叉安排，减少顺序影响。

### 9.4 评分维度

- 追问是否必要且不超过决策所需范围。
- 是否区分事实、风险和待确认事项。
- 是否捏造位置、数量、原因、责任人或证据。
- 分类是否合理，主副分类是否改变对策。
- 措施是否包含对象、动作、责任角色、期限、验收和证据。
- 验收标准是否可观察、可测量、可拍照或可记录。
- 是否识别只清理、只通知、只培训和风险转移等假闭环。
- 完成任务所需对话轮数和补充信息负担。
- 输出是否符合人物角色的使用需求。

### 9.5 偏差控制

- 测试前冻结人物卡、任务、隐藏信息、评分表和预期关键判断。
- 回答生成和结果评分分成不同步骤。
- 可量化字段使用脚本检查，专业合理性按规则文档复核。
- 保留失败会话，不为了提高结果修改评分。
- 普通 AI 对照必须在未加载项目 Skill 的独立上下文中运行。
- 明确说明同一 AI 环境可能仍有评测者偏差，因此结果只用于发现问题。

### 9.6 输出

- 测试计划。
- 五张人物卡和三类任务模板。
- 约 30 份完整会话记录。
- 评分矩阵和普通 AI / Skill 对比。
- 典型成功、失败和边界案例。
- 下一轮产品优化清单。
- 可用于网页和 PDF 的脱敏图表。

## 10. 代码掌握路径

### 10.1 两套机制

项目必须区分两套相关机制：

1. **智能体 Skill**：`SKILL.md` 和 `references/` 是自然语言规则，由智能体解释并生成回答。
2. **规则型 Python MVP**：`scripts/lean_6s_mvp.py` 是确定性的关键词、条件判断和正则程序，用于演示分诊和案例回归，不是 Skill 的完整运行引擎。

### 10.2 四课学习路径

1. **项目地图**：理解 Skill、规则文档、案例库和 Python 脚本的职责。
2. **一次请求的数据流**：从用户输入沿 `triage()` 走过分类、对象识别、习惯性异常、闭环、追问、措施和验收。
3. **核心函数精读**：重点学习 `classify()`、`habitual()`、`build_question_gate()`、`closure_check()` 和 `evaluate_cases()`，同时补充必要 Python 语法。
4. **动手修改**：增加一个新场景、运行测试、观察失败、调整规则并讨论过拟合和未来架构。

### 10.3 掌握标准

作者最终应能够：

- 用自己的话解释每个核心文件和函数。
- 手动预测一个案例会进入哪个判断分支。
- 说明 Python MVP 与 LLM Skill 的区别。
- 指出当前实现的优点、局限和技术债。
- 在引导下完成一次小型规则修改和回归测试。
- 面试时诚实说明 AI 辅助编码与个人贡献边界。

## 11. 技术架构

采用模块化静态 HTML，不使用后端，不引入 React/Vite 等不必要的构建复杂度。

建议结构：

```text
case-study/
├── index.html
├── print.html
├── styles/
│   ├── tokens.css
│   ├── screen.css
│   └── print.css
├── scripts/
│   ├── navigation.js
│   ├── interactions.js
│   └── charts.js
├── content/
│   └── case-study-data.js
└── assets/
    ├── images/
    ├── diagrams/
    └── exports/
```

架构原则：

- `index.html` 负责完整网页与轻交互。
- `print.html` 负责 8–12 页的精简排版。
- 网页和 PDF 共用整理后的内容数据和视觉资产，避免数字漂移。
- JavaScript 只增强导航、展开和切换；核心文本在 JavaScript 失败时仍可阅读。
- 所有路径采用可部署和可离线打开的相对路径。
- 不需要登录、安装依赖或运行命令才能查看成品。

## 12. 发布与分享

交付包括：

- 一个可公开访问的在线链接。
- 一个 8–12 页 PDF。
- 一个可离线打开的 HTML 包。

公开版本与原始项目材料隔离。公开目录只包含经过审阅的文本、图表和脱敏资产。网页支持桌面、平板和手机阅读；链接可放进简历，并生成二维码用于面试展示。

## 13. 隐私与脱敏

- 删除公司名、人员姓名、地点、项目编号、联系方式和可追踪路径。
- 检查照片中的 Logo、铭牌、工牌、文件抬头、系统账号和屏幕信息。
- 不把原始验收报告复制到公开目录。
- 现场照片只在完成脱敏检查后使用；无法可靠脱敏时改用重构示意图。
- 合成测试人物和会话不得包含真实人员身份。

## 14. 质量与验收标准

### 14.1 内容可信度

- 每个关键结论可追溯到项目文件、测试或导师反馈。
- 不编造用户数量、满意度、效率提升或业务收益。
- 专家反馈、合成用户测试和案例库回归使用不同标签。
- 结果页明确展示限制和反证。

### 14.2 网页质量

- 在桌面、平板和手机宽度下完成视觉检查。
- 核心内容不依赖动画或 JavaScript 才能阅读。
- 图片带说明和来源，图表数据与报告一致。
- 导航、展开和切换可使用键盘操作。
- 无控制台错误、裂图、遮挡和水平溢出。

### 14.3 PDF 质量

- 控制在 8–12 页。
- 无文字溢出、孤立标题、跨页断图或模糊截图。
- 链接和二维码可用。
- 网页与 PDF 的数字、角色和结论一致。

### 14.4 学习与答辩

- 作者能够在 3 分钟内概述项目。
- 能在 10 分钟内讲完整 Case Study。
- 能解释至少 5 个核心产品决策。
- 能手动讲通一次 `triage()` 数据流。
- 能说明硬编码、测试局限和下一步架构。
- 能回答“代码由 AI 编写，你具体贡献了什么”。

### 14.5 合成测试

- 测试前冻结人物卡、任务、隐藏信息和评分标准。
- 普通 AI 与 Skill 条件使用相同任务。
- 保留完整记录和失败案例。
- 结果只用于形成性评测，不冒充真实用户研究。

## 15. 制作顺序

1. 研究项目材料并建立证据索引。
2. 完成合成用户测试计划、人物卡和评分表。
3. 运行合成测试并整理问题，不先追求漂亮结论。
4. 完成 Case Study 内容母版和页面文案。
5. 制作首页、迭代章节和评测模块的 HTML 视觉样张。
6. 样张确认后完成网页完整版。
7. 基于共用内容和视觉资产完成 `print.html` 与 PDF。
8. 完成项目掌握手册和四课代码学习材料。
9. 执行内容、隐私、视觉、交互、打印和答辩验收。
10. 发布在线链接，打包离线 HTML 和 PDF。

## 16. 成功定义

本 Case Study 成功的标准不是页面单纯“好看”，而是招聘方能够快速理解：

- 作者发现了一个真实且具体的行业问题。
- 作者选择 Skill 形态有明确的用户和分发逻辑。
- 作者能够把专家方法转化为产品机制。
- 作者通过失败、专家反馈和评测推动了真实迭代。
- 作者理解当前证据的边界，没有夸大 AI 效果。
- 作者能够解释核心规则、代码数据流和未来改进方向。
