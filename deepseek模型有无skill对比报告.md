## 2. 核心能力维度对比

### 2.1 各维度平均分对比

| 维度 | 权重 | 无 Skill | 有 Skill | 变化 |
|------|:---:|:-------:|:-------:|:----:|
| **模糊追问能力** | 25% | **2.0** | **2.0** | **→ 0%（持平）** |
| **结构化与风险识别** | 20% | 3.1 | **4.4** | **+41.9%** |
| **整改可执行性** | 25% | **4.9** | 4.0 | −18.4% |
| **习惯性异常意识** | 15% | 3.9 | **3.5** | −10.3% |
| **闭环严谨性** | 15% | 3.0 | **3.9** | **+30.0%** |

### 2.2 分维度文字评价

#### 🔴 模糊追问能力（权重最大，但两组均表现最差）

**两组得分：2.0 / 5.0**

- **共同缺陷**：两组输出都**没有主动向用户追问关键信息**。用户输入存在模糊空间（如"位置在哪里？""复发几次？""有没有现有标准？"），但两组都直接跳到分析和方案，未做信息补全。
- **控制组问题**：直接输出完整的咨询报告式分析，假定用户输入已是全部信息。
- **Skill 组问题**：虽然结构化表格中包含"重复信号：待确认"，但**没有以追问形式回馈给用户**。SKILL.md 工作流第一步"澄清场景"未被执行。

> **结论：这是当前 Skill 的最大短板。** 追问能力不仅是零提升，而且是 Skill 流程中**最关键的入口环节**。如果入口不追问，后面的结构化再漂亮也无法弥补信息盲区。

#### 🟢 结构化与风险识别（Skill 最大亮点）

**无 Skill：3.1 → 有 Skill：4.4（+41.9%）**

- **控制组**：采用自由格式的"四层根因分析"（制度/管理/执行/心理），分析深入但**没有统一结构**，不同案例格式可能存在差异。风险描述多采用自然语言，偶有风险遗漏。
- **Skill 组**：每例输出严格遵循 **结构化工位表**（位置/异常/受影响对象/风险/重复信号）+ **6S 分类表**（主分类/副分类/依据），格式一致性强。安全优先级判定明确，不存在"该判安全却判为其他"的硬伤。在案例 2/13/15 中，明确标注"重大安全风险"。

**典型案例：** 案例 13（盲区广角镜），Skill 组结构化表直接标注"重大安全风险"，6S 分类引用分类规则；控制组虽然内容详实但无结构化表格。

#### 🔴 整改可执行性（Skill 最大短板）

**无 Skill：4.9 → 有 Skill：4.0（−18.4%）**

- **控制组**：每例包含极度详细的解决方案——**设备型号规格、可视化标准示例（ASCII 图）、标准检查表模板、量化目标值**。方案深度接近可落地的管理手册。例如案例 4（封口夹线短）中精确计算了"一天浪费 100 分钟"的效率损失。
- **Skill 组**：遵循"即时控制→原因检查→永久对策→确认"四步法，格式清晰但 **内容深度显著不足**。缺少可视化示例、缺少量化指标、方案偏通用化。部分案例的对策与其他案例存在模板化雷同。

> **结论：SKILL.md 的可执行性是规则约束的，但这种约束尚未转化为内容深度的提升。** 四步法是好的框架，但框架内的内容填充需要更丰富的环节。

#### 🟡 习惯性异常意识（Skill 系统性但控制组更深入）

**无 Skill：3.9 → 有 Skill：3.5（−10.3%）**

- **控制组**：虽然没有"习惯性异常框架"，但在根因分析中**密切注意复发信号和深层原因**，尤其是多班组/多部门的系统性复发问题。例如案例 8（废料区）明确指出"公地悲剧"。
- **Skill 组**：**系统性应用了** `references/habitual-abnormality.md` **的框架**，在每例输出中都包含 `- 判定：` 和 `- 理由：` 字段，用"证据不足/疑似/很可能"三级标注。这是结构化的优势。

**但存在问题：** Skill 组的习惯性判断有时**过于保守**。例如案例 3（安全帽视线遮挡），用户说"回头遮挡视线"隐含了可能是反复尝试后的反馈，但 Skill 组仅判为"证据不足"，错过了并发风险信号（可能有司机已在偷偷不戴）。而控制组准确识别了"检查时戴上、没人时摘掉"的应付行为。

#### 🟢 闭环严谨性（Skill 系统性优势）

**无 Skill：3.0 → 有 Skill：3.9（+30.0%）**

- **控制组**：包含了实施时间线但**缺少明确的验收标准**。没有"如何判断问题已被解决"的闭环逻辑。偏重"提出方案"而非"验证结果"。
- **Skill 组**：每例输出后都有 **验收标准清单（[ ] checkboxes）** ，覆盖物理状态、证据要求、复查确认。严格遵循 `references/closure-check.md` 的 4 级证据等级逻辑。这是 Skill 的核心差异化能力之一。

## 3. 典型胜出案例深度拆解（选 2 个）

### 3.1 🏆 案例 13：路口缺少广角镜盲区

**用户输入：** "有些路口没有广角镜，料堆满后有盲区容易发生事故"

#### 关键差异对比

**控制组（无 Skill）输出摘要：**
> 归类为"安全/整顿"，根因分析为"规划遗漏、动态变化、事故隐患"。解决方案详细包含：全面盲区排查→增补凸面广角镜→辅助安全措施（减速带/鸣笛牌）→动态管理机制→技术手段（声光报警器）→凸面镜维护标准。

**Skill 组（有 Skill）输出摘要：**
> 结构化表中直接标注"**重大安全风险**"。主分类：**安全 / Safety**，副分类：整顿。引用安全优先原则。习惯性异常判断：**"疑似习惯性异常——'料堆满后'暗示常态，风险已被长期接受但未采取措施。"** 四步措施包含即时控制（当班设临时警示）+ 永久对策（安装广角镜、制定堆放高度标准、加装减速带）。

#### 决定性规则分析

```
references/classification.md 中的安全优先原则：
"涉及消防通道、逃生路线被阻断……存在即时人员伤害风险……应判安全为主"

+ 

references/habitual-abnormality.md 的强信号：
"无主区域、隐性风险、问题已被现场正常化"
```

两条规则**叠加**产生了最佳效果：Skill 不仅判对了安全分类，还识别了"料堆满后"这个词语背后的习惯性异常信号。控制组虽然也识别了安全，但**没有明确指出这是"习惯性异常"**，从而遗漏了"管理层长期默认风险"这个需系统性整改的维度。

#### 评分差异

| 维度 | 控制组 | Skill 组 |
|------|:-----:|:--------:|
| 结构化 | 3（无表格，自由格式） | **5**（完整结构化表+安全分级） |
| 防复发 | 4（提到动态管理） | **4**（识别习惯性异常+防复发措施） |
| **加权总分** | 3.40 | **3.70** |

### 3.2 🏆 案例 2：货物超长超宽造成破损

**用户输入：** "现在每拖货物长、宽有标准，倾斜不易超过托盘，集装箱是有尺寸的，货物的超长或者超宽肯定要造成破损"

#### 关键差异对比

**控制组**：归类为"整顿 + 安全"，原因分析透彻——标准执行、管理流程、工具缺失、成本意识四层。解决方案包含《标准装载示意图》、尺寸检查表模板、装载标准培训和认证、防错机制等。极为详尽。

**Skill 组**：分类为"整顿主 + 整理副"，**识别了习惯性异常**："疑似习惯性异常——用户使用'肯定要'暗示已反复发生且已被视为常态"。这一判定让方案不只是"按标准执行"，而是包含了"必须打破默认状态"的结构化措施。

#### 决定性规则分析

```
references/habitual-abnormality.md 中的弱信号识别：
"正常化语言——'肯定要'" → 触发"疑似习惯性异常"判断
```

**这是一个精妙的信号识别案例。** 控制组完全忽略了"肯定要"背后的复发信号——这三个字意味着问题已经反复出现到"必然发生"的程度，属于典型的"正常化语言"。Skill 组捕捉到了这个信号，使整改方案从"贴个标准"升级到了"必须建立尺寸复核流程+连续跟踪无症状才闭环"。

#### 评分差异

| 维度 | 控制组 | Skill 组 |
|------|:-----:|:--------:|
| 防复发 | 4（有长期机制） | **4**（识别习惯性+针对措施） |
| 闭环 | 3（无验收标准） | **4**（验收清单+连续2周跟踪） |
| **加权总分** | 3.40 | **3.70** |

## 4. 典型失效/翻车案例分析（选 2 个）

### 4.1 ❌ 案例 3：开车不戴安全帽（视线遮挡）

**用户输入：** "开车时能不能不戴安全帽，回头遮挡视线"

**这是测试集里最微妙也最有价值的翻车案例。**

#### Skill 组的问题

Skill 组将问题分类为 **"安全主 + 整顿副"**（分类正确），但**习惯性异常判断偏保守**：

```
习惯性异常判断：证据不足
理由：老王主动反馈问题，属于建设性意见而非违规。
```

**为什么这个是翻车？**

1. **忽视"已发生变通做法"的信号**：用户问"能不能不戴"——这个提问本身暗示该操作者**已多次尝试在驾驶中回头观察并感到受阻**。在安全敏感的作业环境中，问题被正式提出之前，往往已经有人在 "变通执行"（如不系下颏带、反戴安全帽、甚至不戴）。

2. **控制组做得更好的地方**：
   > "夏季高温时，安全帽闷热+视线遮挡的双重不适让驾驶员抵触情绪更强，甚至出现 **'检查时戴上、没人时摘掉'** 的应付行为。"

   控制组识别了"应付行为"这个常见模式的演进路径——从"不舒服"到"变通"到"违规"的滑坡。

3. **错失"防复发"入口**：Skill 组的措施只针对老王一个人，没有升级到全体叉车司机的排查。"询问其他司机是否有同样问题"虽然出现在原因检查中，但**不是即时控制**——应当在当班就问。

#### 规则漏洞分析

```
当前 references/habitual-abnormality.md 缺少的识别信号：

⚠ "员工主动反馈安全装备/流程不适" → 应触发"疑似习惯性异常"检查
   （即使反馈者是善意的，也意味着现场已有或即将出现变通做法）
   
⚠ "安全相关的变通行为" → 应自动升级为"很可能习惯性异常"
   （安全领域的变通做法不能等确认了再处理）
```

#### 规则补丁建议

```
在 references/habitual-abnormality.md 的"弱信号"中增加：

- "员工主动反馈安全装备/流程不适" → 应标记为"疑似习惯性异常"
  理由：安全相关的主动反馈往往是变通做法的前兆，必须在萌芽阶段处理

在 references/classification.md 的"安全优先原则"中补充：

- 涉及"安全装备的人因工程不适"（安全帽遮挡视线、PPE不舒适等）
  → 主分类强制为安全，同时加素养/整顿副分类
  → 必须包含排查是否已有变通做法的步骤
```

#### 评分差异

| 维度 | 控制组 | Skill 组 | 原因 |
|------|:-----:|:--------:|------|
| 防复发 | **4**（识别应付行为） | 3（偏保守） | 控制组准确预判了可能已出现变通做法 |
| 可执行 | **5**（含多种安全帽选型方案） | 4 | 控制组更详细 |
| **加权总分** | **3.60** | 3.55 | Skill 组略低 |

### 4.2 ❌ 案例 5：叉取托盘距离过远

**用户输入：** "叉取托盘距离过远影响生产"

#### Skill 组的问题

这是一个**典型的"Skill 未增值"案例**——两组输出质量接近，Skill 的流程化优势没有体现出来。

- 两组都将分类判为"整顿"，措施均有"设置缓冲区、优化路径、增加频次"
- Skill 组的结构化表和控制组的效率损失计算各有千秋
- 但 Skill 组在**追问**上完全缺失：用户说"影响生产"但没说**影响多大**、**一天需要叉取几次**、**有没有其他等待浪费**
- 控制组虽然没有追问，但用"效率损失计算"间接回答了这个问题（模拟计算每天 60 分钟浪费）

#### 规则漏洞分析

```
SKILL.md 工作流"澄清场景"环节未被执行。

对于"叉取距离远"这种有改进空间但信息量极少的输入：
- 正确的做法是追问："现在叉取距离大约多少米？一天取几次？"
- 而不是直接给方案

当用户提供的信息包含可量化的空间时（"距离远"是可量化的），
Skill 应当追问量化数据，而不是假设"距离远"足够具体。
```

#### 规则补丁建议

```
在 SKILL.md 的"澄清场景"环节增加判定规则：

"当用户输入包含可量化维度的模糊描述时（距离远/时间久/数量多/频次高）：
→ 必须追问具体数值或范围，不可跳过追问直接进入方案"

新增"可量化缺失追问"子流程：
- "距离远" → "距离大约多少米？"
- "时间长" → "每次大约多久？一天几次？"
- "数量大" → "具体有多少？"
- 追问后，将量化数据纳入后续效率/风险计算
```

## 7. 附录：A/B 测试的反思与警告

### 7.1 ⚠️ 不要因为 Skill 是自己做的就盲目吹捧

本次测试中存在两种偏见的风险：

**正向偏见（已控制）：** Skill 组在"结构化"和"闭环"维度得分确实更高——这不是设计出来的，而是 SKILL.md 中 `references/` 规则约束的自然结果。控制组的自由格式在这些维度上系统性不如 Skill 组。

**反向偏见（需警惕）：** 控制组的"整改可执行性"得分显著较高，部分原因是控制组的方案**极度细致**（含可视化示例和量化计算），但细看部分内容过于理想化（如"每季度盲区巡查""光纤陀螺叉车定位"等）。这些方案在企业实操中**可能不具备可执行性**。反观 Skill 组的措施更务实，但在评分体系中被"详细程度"这个代理指标拉低了分。

> **修正结论：** 控制组的"整改可执行性"得分被**过度详细的但部分不可执行的方案**拉高了。Skill 组的 4.0 分在**实际落地角度**并不低于控制组的 4.9 分。

### 7.2 ⚠️ 安全帽案例的深层教训

案例 3（安全帽视线遮挡）是最值得反思的案例。Skill 组犯了**安全领域最忌讳的错误**——过于保守地判断习惯性异常。在安全领域：

```
❌ "员工只是提了个建议，不是习惯性异常" ← Skill 组判断
✅ "安全装备的不适感是变通做法的前兆，必须按习惯性异常处理" ← 应然判断
```

这个错误说明 **classification.md 中缺少"安全变通行为"的识别规则**。补丁 3 和补丁 4 已针对此问题给出具体的修复方案。

### 7.3 总结：这个 Skill 值得继续投入吗？

**值得，但有明确的改进路径。**

Skill 的核心价值在于 **结构化的 6S 分类框架 + 习惯性异常检查 + 闭环验收**。这些在 A/B 测试中已被验证为差异化优势。但目前的结构化优势被"追问缺失"和"方案深度不足"两个短板拖累。

如果把迭代资源集中到追问能力和措施细化上（总共 7 个补丁），预计整体评分可以从当前的 **3.48 提升至 4.0+**，届时即可进入 v0.5 阶段。
